# POP v0.1.8 Option Tree

## Option A

### Goal

Continue lowering LangChain adoption friction.

### What changes

- shorter installed-package-first examples
- clearer evaluation steps for external maintainers
- tighter backward-compatibility guidance for legacy helper names

### What does not change

- no new schema baseline
- no change to the canonical top-level persona boundary
- no major expansion of the helper surface

### Advantages

- keeps the current LangChain line easy to evaluate
- reduces first-contact confusion
- strengthens the current maintainer-facing narrative

### Risks

- may yield limited new technical evidence
- may look incremental if overextended

### Recommended timing

Near-term, if external feedback mainly points to clarity and adoption
friction rather than runtime-fit problems.

### Narrative impact

Package version only, with limited new public narrative emphasis.

## Option B

### Goal

Deepen LangChain execution contract and optional-dependency fit.

### What changes

- stronger installed-package contract tests
- tighter contract semantics around the primary LangChain entry point
- clearer mapping between execution bundle layers and LangChain v1
  evaluation expectations

### What does not change

- no new schema baseline
- no change to the canonical persona-object boundary
- no claim of official LangChain runtime integration

### Advantages

- strengthens the strongest current evidence line
- keeps POP aligned with its current primary runtime target
- improves external review quality without expanding schema scope

### Risks

- can drift into internal refinement without enough visible adoption
- may require more maintainer feedback before the right surface is clear

### Recommended timing

Recommended if feedback suggests the main challenge is not adoption
clarity but execution-contract fitness.

### Narrative impact

Package version plus a moderate new public narrative emphasis around
LangChain contract stability.

## Option C

### Goal

Cautiously elevate CrewAI from a secondary line to a more explicit
runtime track.

### What changes

- more explicit CrewAI-facing documentation
- stronger CrewAI contract tests
- clearer compatibility guidance for Python version constraints

### What does not change

- no new schema baseline
- no change to the canonical persona boundary
- no attempt to make CrewAI the primary runtime line immediately

### Advantages

- broadens runtime relevance
- tests whether POP can sustain a second runtime narrative without
  changing schema scope

### Risks

- can dilute the current LangChain-first story
- may introduce more environmental friction because CrewAI compatibility
  is more constrained
- may split maintainer-facing focus too early

### Recommended timing

Only after LangChain-facing feedback has stabilized enough to justify a
secondary runtime line.

### Narrative impact

Would likely justify both a package version bump and a more visible
public narrative emphasis.

## Recommended Next Step

Option B is currently the most defensible next step. It keeps POP
focused on its strongest runtime line, preserves the canonical schema
baseline at `v0.1.0`, and converts current external feedback into a more
stable LangChain-facing contract surface before any broader runtime
expansion.
